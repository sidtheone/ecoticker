# Monkey Finding — Frame Phase

- **Technique:** Assumption Flip
- **Target:** Frame triage decision — "Medium" classification and Task 5 sequencing: tests run AFTER all implementation (Sentinel writes/updates test contracts failing, then Shipwright implements, then tests pass)
- **Confidence:** 75
- **Survived:** Yes
- **Observation:** The frame assumes Task 5 (test updates) depends on Tasks 1-4 completing first. The dependency diagram literally shows `Task 1-4 ──> Task 5 (tests)`. Flip that assumption: what if tests should be updated FIRST, before implementation? CLAUDE.md line 8 says "TDD. Write tests first, then implement." The plan calls for Sentinel to "write/update test contracts (failing)" as Step 1, but the frame's task table assigns Sentinel only to "Update class assertions, testids, copy" — mechanical fixups AFTER the DOM changes, not red-phase contracts that define what the new DOM SHOULD look like. The frame is describing test-after, not test-first. Specifically: `tests/TopicDetail-7-5.test.tsx` currently asserts `screen.getByTestId("detail-score")).toHaveClass("text-4xl")` at line 339. The plan says the score becomes `text-[72px]` or `text-[48px]` depending on value. That test assertion needs to change BEFORE the implementation, not after — it's the contract that defines the new behavior. Same for the 36+ dimension testid assertions that will change from `dimension-card-*` to `dimension-row-*`. The frame says Sentinel goes first in the sequence (Step 1) but describes Sentinel's scope as "Updates Needed" — past tense, reactive. If Sentinel truly writes failing test contracts first (as the sequence says), then the "Test Files" table should describe NEW assertions (the flat row classes, the giant score sizing, the removed h3), not just "update selectors." The frame's sequence and its task descriptions contradict each other. One of them is wrong. If the sequence is right (Sentinel first, tests-before), the Sentinel scope is under-specified. If the task descriptions are right (update selectors after DOM changes), the sequence violates TDD.
- **Consequence:** If ignored, the team follows the task descriptions (test-after) despite the sequence diagram saying test-first, violating CLAUDE.md's TDD mandate. This means the Shipwright implements Tasks 1-4 against no contract — any deviation from the plan's visual spec won't be caught until someone eyeballs it. Worse: if the Shipwright changes a testid (e.g., `dimension-card-eco` to `dimension-row-eco`) but Sentinel's "update" pass misses one reference in `TopicDetail.test.tsx` line 155, you get a false-green (old test deleted) or false-red (stale selector). Either way, the test suite stops being a reliable contract for the redesign. Fix: resolve the contradiction explicitly — either Sentinel writes red-phase tests FIRST with the new assertions (correct per TDD and CLAUDE.md), or acknowledge this is a refactor-only pass where test-after is acceptable and document why TDD is waived.
