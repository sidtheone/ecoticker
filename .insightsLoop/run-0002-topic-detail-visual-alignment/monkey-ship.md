# Monkey Finding — Ship Step

- Technique: Time Travel
- Target: `src/app/api/topics/[slug]/route.ts` line 19 (`asc(scoreHistory.recordedAt)`) + `src/lib/batch-pipeline.ts` line 1194 (score_history insert with no explicit `recordedAt`) + `src/app/topic/[slug]/page.tsx` line 124 (`scoreHistory[scoreHistory.length - 1]`)
- Confidence: 86
- Survived: yes
- Observation: The `score_history.recorded_at` column is a `date` type (not `timestamp`), and `defaultNow()` truncates to the calendar date. The batch pipeline insert at line 1194 never sets `recordedAt` explicitly. There is no uniqueness constraint on `(topic_id, recorded_at)`. When the batch fails at 3am and an operator re-runs it the same day, two score_history rows are created with the identical `recorded_at` date. The API route orders by `asc(scoreHistory.recordedAt)`, but PostgreSQL's sort order for rows with identical sort keys is non-deterministic. The client page blindly takes `scoreHistory[scoreHistory.length - 1]` as "latest" for dimension scores, insight lede, and sub-score breakdown. On a same-day re-run, the page may render the FIRST (failed/stale) batch's dimension data instead of the corrected re-run's data. No test covers duplicate-date score history entries. Storm, Cartographer, and Editor all examined the page.tsx rendering but none traced the data path back through the API ordering guarantee to the schema's date-vs-timestamp decision.
- Consequence: After any same-day batch re-run (manual recovery, cron retry, operational fix), the topic detail page has a 50/50 chance of showing stale dimension scores, wrong severity levels, and outdated reasoning text to users. The data literally lies — violating "The data hurts. That's the point." because it shows the wrong data silently. The fix is either: (a) add a secondary sort on `id` DESC in the API query to guarantee the latest insert wins, or (b) change `recorded_at` from `date` to `timestamp` so same-day entries have distinct sort keys, or (c) add a UNIQUE constraint on `(topic_id, recorded_at)` with ON CONFLICT DO UPDATE to ensure one entry per topic per day.
