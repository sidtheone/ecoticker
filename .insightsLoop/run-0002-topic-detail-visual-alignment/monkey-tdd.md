# Monkey Finding — TDD Step

- Technique: Hostile Input
- Target: `tests/ArticleList.test.tsx` — all article URL fixtures use well-formed `https://` strings. No test probes `javascript:`, `data:`, or empty-string URLs rendered as `<a href>` targets.
- Confidence: 79
- Survived: pending
- Observation: Every test file across all four suites (`TopicDetail.test.tsx`, `TopicDetail-7-5.test.tsx`, `ArticleList.test.tsx`, `ScoreChart.test.tsx`) treats article URLs as trusted, well-formed strings. The ArticleList component renders them directly into `<a href={url} target="_blank">`. The Sentinel's 23 new tests define the flat-row contract for article rendering (AC5: source citations, no card wrappers, prominent dates) but never inject a hostile URL like `javascript:alert(1)` or `data:text/html,<script>alert(1)</script>`. The plan spec moves articles to tighter, more prominent flat rows — the URL is still the click target. If the implementation does not sanitize `href` values, a poisoned article URL from the GNews API or a malicious RSS feed passes straight through to the rendered DOM as a clickable XSS link. The test suite will green-light it because no assertion checks URL protocol safety.
- Consequence: A single poisoned article URL from an external news source (GNews, RSS) becomes a clickable XSS vector on the topic detail page. The redesign amplifies this because flat rows make article links more visually prominent and clickable. The test suite provides zero protection — every URL fixture is `https://example.com/*`. Without a test asserting that non-http(s) URLs are either stripped, blocked, or rendered as plain text, the contract allows the implementation to pass hostile URLs straight to the user's browser. VALUES.md says "Validate at the door" — the door here is the rendering layer, and no test guards it.
