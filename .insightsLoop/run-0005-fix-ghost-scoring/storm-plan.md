## Introduced Issues

| Location | Issue | Severity | Suggestion |
|----------|-------|----------|------------|
| Architecture #3 + #5 | **articleCount double-fix creates a correctness gap.** The plan fixes both line 1166 (`topics.articleCount + topicArticles.length` -> `+ newArticles.length`) and line 1190 (`articleCount++` guarded by `!existingUrlSet.has`). But these are counting different things. Line 1166 is the *database column* (`topics.article_count`), accumulated across all runs. Line 1190 is the *pipeline return counter* (`articlesAdded`), tracking this run only. The plan conflates them by calling both "articleCount" but they have different semantics. **Scenario:** First run inserts 5 articles. `topics.article_count = 5`, `articlesAdded = 5`. Second run fetches 3 new + 2 dupes. Plan says line 1166 becomes `+ newArticles.length` (3), so DB gets `5 + 3 = 8`. Correct. But the insert loop still iterates ALL `topicArticles` (5), calling `onConflictDoNothing` for the 2 dupes. The guard at line 1190 uses `existingUrlSet` to skip those 2, so `articlesAdded = 3`. Also correct. **However:** the insert loop still calls `db.insert()` for articles that are known dupes ŌĆö unnecessary DB round-trips that the skip guard at the top already identified. This is wasted work, not a correctness bug, but it contradicts the plan's key decision bullet that says "Article insert loop still iterates ALL topicArticles (onConflictDoNothing handles dupes)" ŌĆö if you already know which are dupes, why insert them? | medium | Either (a) iterate only `newArticles` in the insert loop (simpler, fewer DB calls, consistent with the skip guard), or (b) drop the line 1190 guard entirely and count `newArticles.length` directly. The plan currently does redundant work by pre-querying dupes but still inserting them. Pick one dedup strategy, not two. |
| Architecture #2 (skip guard) | **Skip guard skips scoring AND score_history ŌĆö plan silent on whether this is desired.** When `newArticles.length === 0`, the `continue` skips the entire loop body: topic upsert, article insert, score_history insert, and keyword insert. The score_history table has same-day dedup via `onConflictDoUpdate`, so a same-day re-run would just update the existing row. With the skip guard, it won't even try. This is correct for preventing ghost scores, but the plan's Intent says nothing about score_history behavior on re-runs. If someone expects the score_history to still record "we checked and nothing changed," that's lost. | low | Add a sentence to the plan's key decisions: "Skip guard means no score_history entry is written for all-dupe topics. This is intentional ŌĆö no new data means no new score." Explicit > implicit. |
| Architecture #2 (pre-query) | **`inArray` with empty array is undefined behavior in some SQL dialects.** If `topicArticles` is empty (theoretically possible if classification maps an article to a topic but the article was filtered), `fetchedUrls` would be empty, and `inArray(articles.url, [])` may produce `WHERE url IN ()` which is a SQL syntax error in PostgreSQL. | medium | Guard: `if (topicArticles.length === 0) continue;` before the pre-query. Or: the plan should state that `topicArticles` is guaranteed non-empty by the `topicGroups` Map construction (only entries with at least one article exist). Make the assumption explicit. |
| Task 1 (TDD) | **No test case for the `inArray` pre-query mock.** The existing mock DB (`createMockDb`) uses a single chain where all `select()` calls share state. The plan adds a new SELECT query (the `inArray` pre-query) inside the per-topic loop, *before* the existing topic upsert. The mock's `selectCallCount` tracking in `mockForDaily()` will break ŌĆö the first select was "existing topics with keywords," but now there's a second select per topic (the URL pre-query) before the topic upsert. | high | The plan's Task 1 says "Write tests for skip-on-all-dupes, partial-dupes-count" but doesn't acknowledge that `mockForDaily()` needs restructuring. Each topic iteration now has its own SELECT, which means the mock must distinguish between "load existing topics" (once) and "check existing URLs" (per-topic). This is the hardest part of the implementation and the plan doesn't address it. |
| Task 6 (correction script) | **`scripts/fix-article-counts.ts` reads `topics.article_count` and compares to `COUNT(articles.id) WHERE topic_id = ?`, but the plan doesn't specify what to do when articleCount is LOWER than actual.** This could happen if topics were seeded with `articleCount: 0` but articles were inserted separately (e.g., via the seed script). The plan only says "corrects inflated counts" but doesn't define the direction of the fix. | medium | Specify: does the correction script fix *only* inflated counts (article_count > actual), or *any* mismatch (article_count != actual)? The latter is safer and more general. State it explicitly. |
| AC #3 vs Architecture | **AC #3 says `articlesAdded` reflects "actual DB inserts," but the plan doesn't make it reflect actual DB inserts.** The plan's fix at line 1190 uses `!existingUrlSet.has(a.url)` to guard the counter, which counts articles that are *expected* to insert (not yet in DB when pre-queried). But if two topics share an article URL (cross-topic dedup), the first topic's insert succeeds, the second topic's insert hits `onConflictDoNothing`. The pre-query only checks articles already in DB *before this batch run*, not articles inserted earlier in the same loop iteration. So `articlesAdded` can still overcount. | high | **Scenario:** Article URL "reuters.com/x" is classified under both "Climate Crisis" and "Energy Policy". First topic inserts it (succeeds, counter++). Second topic pre-queries ŌĆö URL not in DB yet at query time? No: the first topic already inserted it in the same transaction. Actually, there's no transaction wrapping the loop ŌĆö each insert is its own await. So the second topic's pre-query WILL find the URL (it was committed by the first topic). This means the pre-query works correctly for cross-topic dedup within a single batch run. **But only because there's no transaction.** The plan should state this assumption: "Pre-query correctness depends on each insert being immediately visible (no wrapping transaction)." If someone later adds a transaction around the loop, the fix breaks silently. |
